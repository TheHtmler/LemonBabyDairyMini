# 食谱功能设计

## 背景

当前项目已经支持两种食物记录方式：

- 在 `daily-feeding` 页直接添加单条食物记录
- 在 `meal-editor` 页把多种食物组合成“一顿饭”

这套模型已经能够表达“某一顿吃了哪些食物、每项多少克或多少毫升”，并且最终统一落到 `feeding_records.intakes[]`。历史页、趋势页、分析页、报告预览页也都围绕 `intakes[].nutrition`、`naturalProtein`、`specialProtein` 这些最终快照字段做统计。

当前缺少的是“可复用的餐次模板”。用户如果经常做同一份搭配，例如：

- `A 10g`
- `B 3ml`
- `C 25g`

每次都需要重新把这三项逐个添加并配置份量，操作重复且容易出错。

本次目标是在不破坏既有历史逻辑的前提下，引入“食谱模板”能力：

- 允许把一组食物和基准份量保存为食谱
- 下次记录时可直接选择该食谱
- 允许输入“本次吃了多少百分比”
- 系统按比例计算每一项食物的最终摄入量和营养值
- 历史展示仍以“实际吃了什么”为主，而不是以模板对象为主

## 目标

- 支持把多种食物及其基准份量保存为可复用食谱模板
- 支持在 `meal-editor` 中直接从食谱添加本顿食物
- 支持给整顿食谱输入统一的食用比例，例如 `70%`
- 支持按比例缩放食谱中每种食物的 `quantity` 与营养结果
- 支持将已记录的一顿饭一键保存为食谱模板
- 支持食谱引用后继续编辑单个食物，仅默认影响本次记录
- 若本次内容与原食谱不一致，保存时提醒用户是否更新模板
- 历史页继续展示真实摄入量，并额外标记“来自某食谱、食用多少比例”
- 保持旧记录、旧报表、旧分析逻辑可用，不要求迁移历史数据

## 非目标

- 不把食谱作为新的历史记录主类型单独存入 `feeding_records`
- 不改写旧的 `feeding_records.intakes[]` 汇总口径
- 不要求一次记录支持“每种食物不同食用比例”
- 不把食谱和食物合并到同一个集合或同一个管理模型里
- 不批量回填历史记录的食谱来源信息
- 不改变奶量记录、药物记录、治疗记录的现有数据模型

## 核心设计原则

### 原则 1：食谱是模板，历史是快照

食谱只负责描述“这顿饭通常怎么做”，历史记录继续描述“这次实际吃了多少”。模板和历史必须分层：

- 模板可修改、删除、迭代
- 历史记录一旦写入，不受模板后续变化影响

### 原则 2：食谱不改变历史存储口径

通过食谱生成的一顿饭，在保存时仍然展开成普通 `intakes[]` 项目。每个项目都拥有最终的：

- `quantity`
- `nutrition`
- `naturalProtein`
- `specialProtein`

现有汇总与报表继续消费这些最终字段，无需递归解析食谱结构。

### 原则 3：模板修改默认不回写

用户从食谱创建本顿后，可以继续修改单个食物份量、删减食物或补充新食物。默认这些改动只影响本次记录，不自动更新模板。只有用户明确确认时，才更新食谱。

### 原则 4：更新模板更新的是基准量

若本次是从某个食谱按 `70%` 生成，再将其中一项从 `17.5g` 改为 `20g`，更新模板时应反推出新的模板基准量 `20 / 0.7 = 28.57g`，而不是直接把模板改成 `20g`。模板永远表示完整做法，比例只属于某次食用。

## 数据模型设计

### 新增集合：`recipe_catalog`

新增食谱模板集合，与现有 `food_catalog`、`feeding_records` 分开。

建议字段：

- `babyUid`
- `name`
- `note`
- `items`
- `items[].foodId`
- `items[].nameSnapshot`
- `items[].category`
- `items[].unit`
- `items[].baseQuantity`
- `items[].nutritionSnapshot`
- `items[].proteinSource`
- `items[].proteinQuality`
- `items[].naturalProtein`
- `items[].specialProtein`
- `createdFrom`
- `createdFrom.mealBatchId`
- `createdFrom.date`
- `createdAt`
- `updatedAt`
- `lastUsedAt`
- `useCount`
- `status`

### 食谱项字段含义

`items[]` 表示食谱模板中的基准组成。每项都保存基准快照：

- `baseQuantity`：该食材在完整做法下的基准量
- `nutritionSnapshot`：该基准量下的营养结果
- `nameSnapshot`：模板保存时的食材名称
- `proteinSource / proteinQuality / naturalProtein / specialProtein`：用于延续现有营养拆分逻辑

这样即使食物库后续变化，模板本身仍可稳定展示。

### 继续沿用：`feeding_records.intakes[]`

历史记录不新增新的主结构，仍然写入 `feeding_records.intakes[]`。但为支持食谱来源展示，需要给每条由食谱展开得到的 intake 增加可选来源字段：

- `sourceType: 'manual' | 'recipe'`
- `recipeId`
- `recipeNameSnapshot`
- `recipeConsumedRatio`
- `recipeBaseQuantity`
- `recipeScaledFromBase: true`

说明：

- `recipeConsumedRatio`：本次使用时填入的整顿比例，例如 `0.7`
- `recipeBaseQuantity`：该食材在模板中的原始基准量
- `quantity`：本次实际摄入量，等于 `recipeBaseQuantity * recipeConsumedRatio`

### 为什么不把食谱直接存进 `intakes[]`

不推荐把食谱存成一条嵌套 intake，例如：

- 一条餐次
- 内部再嵌套多条食材

因为当前 `daily-feeding`、`data-records`、`trend-report`、`data-analysis`、`reportWorkbench` 都默认 `intakes[]` 是最终可汇总的平面结构。若改成嵌套结构，需要同步改动所有统计与展示路径，风险远大于收益。

## 记录口径

### 食谱模板口径

食谱模板描述的是完整做法，不包含“这次吃了多少”的状态。

例如：

- `A 10g`
- `B 3ml`
- `C 25g`

这就是模板本身。

### 本次记录口径

如果本次输入 `70%`，则保存时展开为：

- `A 7g`
- `B 2.1ml`
- `C 17.5g`

每条 intake 的 `nutrition / naturalProtein / specialProtein` 也按最终摄入量重新计算并落库。

### 更新模板口径

如果本次是按 `70%` 展开后又手动修改：

- `A 7g`
- `B 2.1ml`
- `C 20g`

那么选择“更新食谱”后，新的模板基准应为：

- `A 10g`
- `B 3ml`
- `C 28.57g`

而不是把模板直接改成：

- `A 7g`
- `B 2.1ml`
- `C 20g`

## 交互设计

### 1. 从食谱添加本顿食物

主入口放在 `meal-editor`，不新开平行的“食谱记录流程”。

推荐流程：

1. 用户进入 `meal-editor`
2. 选择“从食谱添加”或继续手动添加食物
3. 选中食谱后，先看到该食谱的基准内容
4. 输入本次食用比例，默认 `100%`
5. 系统将每一项食材按比例展开到当前 `mealDraft.items`
6. 用户可继续删改单个食物

这样所有“这顿饭怎么组成”的操作仍然集中在一个页面里。

### 2. 食谱管理页

新增轻量页面，例如：

- `pages/recipe-management/index`

职责只做：

- 食谱列表
- 新建食谱
- 编辑食谱
- 删除食谱

不要让该页面承担“当天记录”职责，避免和 `meal-editor` 混用。

### 3. 从一顿饭保存成食谱

在 `meal-editor` 编辑已存在餐次时，增加“存为食谱”动作。

推荐流程：

1. 打开一顿已有餐次
2. 点击“存为食谱”
3. 填写食谱名称和可选备注
4. 以当前餐次的完整内容作为模板基准量保存

默认保存的是“当前编辑后的完整做法”，不携带本次食用比例。

### 4. 食谱引用后的临时改动

默认行为：

- 改动只影响本次记录
- 模板保持不变

保存时若检测到当前内容与原食谱不一致，弹出提示：

`这次内容和食谱【xxx】不同，是否更新该食谱？`

建议操作顺序：

- `仅本次`
- `更新食谱`

### 5. 历史展示

按确认过的方案，历史仍按普通餐次展开显示：

- 先展示这顿饭的时间、名称、汇总营养
- 再展开显示每种食物的真实摄入量

仅增加轻量标签：

- `来自食谱：xx`
- `食用 70%`

不把历史页改造成“食谱卡片优先”的结构。

## 页面职责拆分

### `meal-editor`

新增职责：

- 选择食谱
- 输入食用比例
- 将食谱按比例展开为 `mealDraft.items`
- 保存时检测当前内容是否偏离原食谱
- 支持“存为食谱”

仍保留原职责：

- 手动添加/编辑食物
- 保存本顿餐次到 `feeding_records.intakes[]`

### `recipe-management`

新增页面职责：

- 查看食谱列表
- 新建 / 编辑 / 删除模板
- 维护模板名称、备注、食材基准量

### `daily-feeding`

只做最小改动：

- 显示餐次来自哪个食谱
- 显示该餐次食用比例

### `data-records`

与 `daily-feeding` 保持一致：

- 继续展示实际 intake
- 透传并展示 recipe 来源信息

## 读写路径设计

### 模板创建

食谱可来自两类入口：

- 在 `recipe-management` 手工新建
- 在 `meal-editor` 中将当前餐次保存为食谱

手工新建时，模板项可直接复用现有食物选择器能力。每项模板保存时应计算并写入对应的基准营养快照。

### 使用食谱

使用食谱时建议流程：

1. 读取模板项 `items[]`
2. 输入统一比例 `ratio`
3. 对每一项执行：
   - `quantity = baseQuantity * ratio`
   - `nutrition = scale(nutritionSnapshot, ratio)` 或按最新食物定义重算
   - `naturalProtein / specialProtein` 同步按比例缩放
4. 生成 `mealDraft.items`
5. 保存时展开写入普通 `intakes[]`

### 保存历史记录

通过食谱生成的餐次，仍然写入：

- `type: 'food'`
- `foodType: 'food'`
- `mealBatchId`
- `mealTime`
- `mealLabel`
- `mealNote`

并额外写入：

- `sourceType: 'recipe'`
- `recipeId`
- `recipeNameSnapshot`
- `recipeConsumedRatio`
- `recipeBaseQuantity`

手动添加的餐次保持：

- `sourceType: 'manual'`

或不写该字段。

## 模板与食物库的关系

### 模板显示

模板列表和模板详情默认优先显示模板自己的快照：

- `nameSnapshot`
- `baseQuantity`
- `nutritionSnapshot`

这样模板内容稳定，不会因为食物库改名或改图而无法展示。

### 模板再次使用

食谱再次被引用时建议采用以下规则：

- 若 `foodId` 仍能在 `food_catalog` 找到，优先按最新食物定义重新计算本次 intake
- 若 `foodId` 找不到，则提示该食材已失效，需要替换或删除后再保存

这样既能利用新版食物营养定义，又不会让模板本身无法读。

## 兼容策略

### 不迁移旧数据

本次功能不需要全量迁移旧记录。

原因：

- 旧记录已经是完整摄入快照
- 新增的食谱来源字段只是可选附加信息
- 回填历史来源信息成本高，收益低

### 旧逻辑继续成立

旧记录没有 recipe 字段：

- 历史展示照常
- 汇总统计照常
- 趋势图照常
- 报告预览照常

新记录虽然来自食谱，但最终也只是普通 intake 快照，因此不会破坏现有计算模型。

### 模板更新不影响历史

更新或删除食谱模板时：

- 不回写历史记录
- 不重算旧餐次
- 历史仍保留 `recipeNameSnapshot`

即使模板后续更名或删除，历史里仍能知道当时来自哪个模板。

## 边界规则

- 食谱模板只保存基准做法，不保存“这次吃了多少”
- 每次使用食谱只允许一个统一比例，不支持每种食材独立比例
- 展开后的餐次仍允许手工补食物、删食物、改单项份量
- 模板更新必须由用户主动确认，不自动回写
- 删除历史餐次不影响模板
- 删除模板不影响历史餐次
- 若模板中的某个 `foodId` 失效，历史正常显示；新引用时要求修正

## 风险

### 风险 1：在某些页面漏展示 recipe 来源信息

当前 `daily-feeding` 和 `data-records` 都维护了自己的餐次分组逻辑。若只改一处，可能导致两个页面展示不一致。

### 风险 2：更新模板时错误覆盖为“本次摄入量”

若保存“更新食谱”时没有按 `ratio` 反推基准量，模板会越来越小，最终失去“完整做法”的意义。

### 风险 3：模板项依赖的食物被删除或修改

如果没有明确的失效处理策略，用户在再次引用模板时可能遇到无法解释的营养偏差或保存失败。

### 风险 4：把食谱和历史记录混在一起

如果在实现时把食谱当成新的历史主对象，而不是展开成最终 intake，会导致 `trend-report`、`data-analysis`、`reportWorkbench` 全部需要重做解析逻辑，风险极高。

## 测试重点

- 手工新建食谱后，再次引用是否能正确生成餐次
- `100% / 70% / 35%` 等比例下，`quantity` 与营养值是否线性缩放
- 从已有餐次“存为食谱”后，再次引用是否能还原完整基准量
- 从食谱生成餐次后改单项份量，是否默认只影响本次
- 保存时是否准确弹出“是否更新食谱”
- 选择“仅本次”后模板是否保持不变
- 选择“更新食谱”后模板是否按反推基准量更新
- `daily-feeding` 是否能显示“来自食谱 + 食用比例”
- `data-records` 是否能显示同样的食谱来源信息
- `trend-report`、`data-analysis`、`reportWorkbench` 对食谱记录的统计是否与手工逐项录入一致
- 旧记录、非食谱记录、仅单项食物记录是否无回归

## 结论

推荐方案可以概括为一句话：

**食谱是模板，历史是快照；模板负责复用录入，历史继续按最终摄入量结算。**

在当前项目结构下，这条路线的优点最明确：

- 与现有 `feeding_records.intakes[]` 模型兼容最好
- 对历史页、趋势页、分析页改动最小
- 不需要迁移旧数据
- 能满足“下次直接添加整顿食谱”和“吃不完按比例计算营养”的核心诉求

后续实现时，应优先保证：

- 模板独立建模
- 历史继续快照落库
- 页面展示只做轻量增强

不要把食谱引入为新的历史主结构。
