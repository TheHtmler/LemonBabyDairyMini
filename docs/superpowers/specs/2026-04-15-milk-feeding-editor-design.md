# 配奶录入新页面设计

## 背景

当前 `daily-feeding` 页的“添加喂奶”主入口是快捷弹窗，面向母乳、普奶、特奶统一录入。对于普奶和特奶，现有交互只有“体积输入 -> 按标准配比反推粉重”的单向模型。

这会导致一个真实场景无法表达：用户希望本次记录的实际加水量与标准配比不同，但实际粉重保持用户手工输入值。例如标准配比为 `5g : 30ml`，用户本次想记录 `40ml 水 + 5g 粉`。当前实现会把 `40ml` 自动换算成 `6.67g`，与真实记录不一致。

本次改动目标是不动现有 `feeding-modal` 组件和当前快捷弹窗链路，新增一个独立页面承载普奶/特奶的“水量 + 粉重”双输入逻辑，并新增一个按钮入口跳转到该页面。

## 目标

- 新增独立页面，处理普奶/特奶录入，不改现有喂奶弹窗。
- 新页面支持“水量”和“粉重”双输入。
- 默认基于标准配比自动换算。
- 支持用户手动修改任一字段，并自动联动另一个字段。
- 支持用户一键恢复到标准配比计算结果。
- 保存口径明确：
  - 摄入体积 = 实际加水量
  - 营养计算 = 优先按实际粉重
  - 不关注混合后成品体积
- 继续写入现有 `feeding_records.feedings` 结构，保证旧数据兼容。

## 非目标

- 不重写现有 `feeding-modal`。
- 不移除现有快捷弹窗入口。
- 不改母乳录入模式。
- 不调整全局营养设置结构。
- 不引入“稀奶/浓奶”作为额外模式名称或业务类型。

## 用户流程

### 入口

在 `daily-feeding` 的“快捷记录”区域新增一个按钮，例如“配奶录入”或“录入普/特奶”，点击后跳转到新页面，交互方式与“添加食物 -> 跳转 meal-editor”一致。

现有“添加喂奶”按钮保持不变，便于并行验证新链路。

### 页面内流程

1. 用户进入新页面。
2. 选择奶种：`普奶` 或 `特奶`。
3. 填写喂奶时间。
4. 在同一区域录入：
   - 水量（ml）
   - 粉重（g）
5. 页面根据当前奶种的标准配比，自动联动另一项输入。
6. 页面实时展示蛋白、热量等营养预览。
7. 用户可点击“恢复标准配比”，按当前字段重新回到标准配比联动结果。
8. 用户确认保存后，写入当天 `feeding_records` 对应文档。

## 交互规则

### 适用范围

- 普奶：支持双输入。
- 特奶：支持双输入。
- 母乳：不进入该页面，也不展示粉重概念。

### 初始状态

- 页面默认时间为当前时间。
- 默认奶种可优先复用最近一次的普奶/特奶选择，没有则默认 `特奶` 或按产品更适合的默认值实现。
- 默认水量和粉重使用最近一次同奶种记录；若没有历史数据，则按标准配比从一个合理的默认水量推导粉重，或直接留空。

### 联动规则

页面维护一个 `lastEditedField`，取值为 `water` 或 `powder`。

- 当用户修改水量时：
  - `lastEditedField = water`
  - 根据标准配比自动重算粉重
- 当用户修改粉重时：
  - `lastEditedField = powder`
  - 根据标准配比自动重算水量
- 当用户切换奶种时：
  - 使用新奶种的标准配比重算另一字段
  - 优先保留 `lastEditedField` 对应字段值

### 恢复标准配比

页面提供“恢复标准配比”按钮。

- 若 `lastEditedField = water`，则按当前水量重算粉重。
- 若 `lastEditedField = powder`，则按当前粉重重算水量。

这个按钮只影响当前页面表单，不修改 `nutritionSettings` 中的标准配比。

### 手动修改的定义

这里的“支持手动修改”不是允许水量和粉重同时各自独立存在而不联动，而是：

- 任意一个字段都允许直接输入
- 系统永远以“最后修改的字段”为主值
- 另一个字段按标准配比自动跟随

这样既满足自由输入，也避免出现两个字段都被手改后互相矛盾、无法确定哪一个是真实主值的问题。

## 数据模型与保存口径

### 主数据源

- 集合：`feeding_records`
- 主关联键：`babyUid`
- 每日一条文档，单次喂奶记录存入 `feedings` 数组

### 保存结构

不新增集合，不改每日文档结构，继续沿用单条 `feeding` 记录对象。

#### 普奶记录

- `naturalMilkType = 'formula'`
- `naturalMilkVolume = 实际加水量`
- `formulaPowderWeight = 实际粉重`
- `specialMilkVolume = 0`
- `specialMilkPowder = 0`

#### 特奶记录

- `naturalMilkType` 保持当前页面约定的默认值，不作为主要信息来源
- `specialMilkVolume = 实际加水量`
- `specialMilkPowder = 实际粉重`

### 总量口径

- `totalVolume` 继续表示本次摄入液体量
- 在新页面中，`totalVolume = 实际加水量`
- 不记录也不推导混合后的成品体积

## 计算规则

### 标准配比换算

设标准配比为：

- `powderRatio = 配方粉重`
- `waterRatio = 标准水量`

则：

- 按水量算粉重：`powder = water * powderRatio / waterRatio`
- 按粉重算水量：`water = powder * waterRatio / powderRatio`

### 摄入统计

- 普奶摄入体积：取 `naturalMilkVolume`
- 特奶摄入体积：取 `specialMilkVolume`
- 汇总中的体积类指标按“实际加水量”累计

### 营养统计

营养计算优先级需要统一升级为：

1. 若记录中存在显式粉重字段，则优先用显式粉重
2. 若不存在显式粉重，再按标准配比由体积反推粉重

应用到具体奶种：

- 普奶：
  - 优先使用 `formulaPowderWeight`
  - 若不存在，再由 `naturalMilkVolume + formula_milk_ratio` 反推
- 特奶：
  - 优先使用 `specialMilkPowder`
  - 若不存在，再由 `specialMilkVolume + special_milk_ratio` 反推

### 对现有逻辑的影响

这意味着当前项目中的奶类计算模型要从“体积主导”调整为：

- 体积用于摄入量统计
- 粉重用于营养统计

旧数据仍兼容，因为旧记录没有显式粉重时仍可回退到现有配比推导逻辑。

## 页面实现建议

### 新页面职责

新增页面建议命名为：

- `pages/milk-feeding-editor/index`

页面职责：

- 读取营养设置
- 读取最近一条同类奶记录作为默认值
- 维护页面级表单状态
- 负责水量/粉重联动
- 负责营养预览
- 保存单条喂奶记录

### 状态建议

- `milkKind`: `formula` | `special`
- `recordTime`
- `waterVolume`
- `powderWeight`
- `lastEditedField`
- `notes`
- `nutritionPreview`
- `ratioDisplay`
- `canRestoreStandardRatio`

### 与现有页面的边界

- `daily-feeding`：只负责新增跳转入口
- `feeding-modal`：不修改
- 新页面自行处理表单和保存

## 写入路径

新页面保存时复用现有“查找或创建当天文档”逻辑，优先复用 `findOrCreateDailyRecord`。

保存流程：

1. 读取 `babyUid`
2. 计算当天 `date key`
3. 通过 `findOrCreateDailyRecord` 找到当天文档
4. 构造 `feeding` 对象
5. `push` 到 `feedings`
6. 刷新返回上一个页面或给出成功反馈

## 读路径与兼容

需要同步检查并修正这些读取/计算路径：

- `daily-feeding` 汇总
- `data-records` 汇总
- 奶类营养预览
- 喂奶记录卡片展示
- 任何按体积反推粉重的工具函数

兼容原则：

- 旧记录没有显式粉重：按当前回退逻辑处理
- 新记录有显式粉重：必须优先使用显式粉重

## 风险

### 风险 1：现有汇总继续按标准配比重算，导致营养统计失真

这是本次改动的核心风险。如果新页面保存了显式粉重，但汇总和分析仍只按标准配比从体积反推，实际结果会被覆盖掉。

### 风险 2：普奶目前缺少统一的显式粉重优先路径

当前项目里特奶已有 `specialMilkPowder` 语义，但普奶是否在所有计算路径中都优先读取 `formulaPowderWeight` 还需统一核查。

### 风险 3：`daily-feeding` 与 `data-records` 有并行实现

喂奶录入和汇总逻辑分散在两个页面中，任何字段语义变化都可能出现一边修复、一边遗漏的问题。

### 风险 4：旧 UI 仍保留旧模型

由于现有快捷弹窗不改，短期内系统会同时存在两套录入模型：

- 旧弹窗：体积主导，粉重自动反推
- 新页面：体积/粉重双输入，营养优先按显式粉重

这在过渡期是允许的，但需要明确这是并行试验状态。

## 测试建议

### 单元测试

- 标准配比下：
  - 改水量能正确反推粉重
  - 改粉重能正确反推水量
- 保存时：
  - 普奶写入 `naturalMilkVolume + formulaPowderWeight`
  - 特奶写入 `specialMilkVolume + specialMilkPowder`
- 计算时：
  - 有显式粉重时优先用显式粉重
  - 无显式粉重时回退标准配比

### 页面级测试

- `daily-feeding` 新按钮可跳转到新页面
- 新页面切换奶种时联动正确
- “恢复标准配比”按钮行为正确
- 保存成功后当天记录列表与汇总正确刷新

### 回归测试

- 现有快捷喂奶弹窗仍可正常保存
- 旧喂奶记录展示不受影响
- `data-records` 页仍能正确读取旧记录和新记录

## 实施建议

建议分两步实施：

1. 新页面 + 新按钮入口 + 新页保存逻辑
2. 统一营养计算优先级，确保显式粉重优先

这样可以先隔离交互风险，再收敛统计口径风险。
